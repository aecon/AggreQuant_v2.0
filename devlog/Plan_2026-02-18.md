### 2. Simplify pipeline image discovery — replace nested search with triplets

**Problem**: The current `pipeline.py` → `ImageLoader` → `group_files_by_field` flow is
convoluted. It groups files by well, then by field, then searches each field's file list
for channel patterns — a search at every loop iteration. Since all results end up in one
CSV, the well→field nesting is unnecessary overhead.

**Goal**: Discover image triplets (nuclei, cells, aggregates) once upfront, then iterate
over them directly. Triplets sorted by (well_id, field_id) for deterministic processing.

#### Proposed changes

1. **New function `build_field_triplets()` in `loaders/images.py`**
   - Input: all image files + channel purpose→pattern mapping
   - Parse every filename once → extract (well_id, field_id, channel_pattern)
   - Group into triplets: `{(well_id, field_id): {purpose: Path, ...}}`
   - Return a list of `FieldTriplet` (NamedTuple with well_id, field_id, and
     purpose→path dict), **sorted by (well_id, field_id)**
   - Skip incomplete triplets (log a warning)

2. **Simplify `pipeline.run()`**
   - Replace `ImageLoader` + nested well/field loops with a single call to
     `build_field_triplets()`
   - Iterate flat sorted list: `for triplet in triplets:`
   - `_process_field` receives the triplet directly (no more per-file search)

3. **Simplify `_process_field()` and remove `_load_channel()`**
   - Receive paths directly from triplet → just call `load_image(path)` for each
   - Delete `_load_channel()` (no longer needed)

4. **Clean up `ImageLoader`**
   - Remove methods no longer used by pipeline (`load_field`, `load_well_channel`,
     `find_channel_files`)
   - Keep `ImageLoader` only if it's used elsewhere; otherwise delete entirely

5. **Verify saving and plotting still work**
   - `_save_results()` depends only on `self._field_results` (list of dicts) — unchanged
   - `_generate_plots()` depends only on the CSV file — unchanged
   - `_save_masks()` uses `well_id` and `field_id` — available from triplet

### 3. Skip segmentation for already-processed fields

**Problem**: Re-running the pipeline re-segments every field, even when label images
already exist on disk. This wastes time when resuming a partially-completed run or
re-processing for updated quantification.

**Goal**: Before segmenting a field, check if its label TIFs already exist. If they do,
load them from disk instead of running segmentation + post-processing. Still compute
focus metrics and quantification so the CSV is always complete.

#### Changes
1. **Add `overwrite_segmentation: false` to `OutputConfig`** in `config.py`
   - YAML flag the user sets to `true` to force re-segmentation
2. **Add `_masks_exist()` and `_load_masks()` helpers** in `pipeline.py`
   - Check / load all 3 label TIFs for a given (well_id, field_id)
3. **Update `_process_field()`** to skip segmentation when masks exist and overwrite=false
4. **Update example config** `configs/test_384well.yaml` with new flag
5. **Tests pass**

### 4. QC strip plot — % aggregate-positive cells by control condition

**Goal**: After the pipeline runs, generate a QC plot that shows how the control wells
behaved. This validates the assay before interpreting sample wells.

#### YAML config — already supports this

The existing `control_wells` dict in the config is already the right structure:

```yaml
control_wells:
  negative:        # condition name (arbitrary, user-defined)
    - A01
    - A02
    - B01
    - B02
  NT:              # another condition
    - A23
    - A24
    - B23
    - B24
```

Users can define as many conditions as they want. Each key is a condition label, each
value is a list of well IDs. **No config changes needed.**

#### How control conditions propagate in the code

`PipelineConfig.control_wells` is already `Dict[str, List[str]]` and parsed from YAML.
The QC module receives this dict plus the field_measurements CSV. Flow:

1. `field_measurements.csv` has per-field rows with `well_id` and
   `pct_aggregate_positive_cells` (and `n_aggregate_positive_cells`, `n_cells`)
2. QC module aggregates to **per-well** values using `compute_ratio_per_well()`
   (already in `heatmaps.py` — reuse or extract to a shared utility)
3. Map each well to its condition using the `control_wells` dict
4. Plot

#### Plot design — strip/swarm plot with summary statistics

```
 %Agg+ │                   ╷
       │    ╷              ┬┤
       │   ┬┤     •       ├┤
       │   ├┤    •••      ┴┤
       │   ┴┤     •        ╵
       │    ╵
       │
       └──────────────────────
          Negative    NT
```

- **X-axis**: control condition names (categorical)
- **Y-axis**: % aggregate-positive cells (per well)
- **Each dot**: one well (jittered horizontally to avoid overlap)
- **Overlay per condition**: mean marker + error bar (std)
- Library: **matplotlib** (already a dependency, keeps it simple; no need for plotly
  interactivity on a QC summary)

#### Implementation plan

1. **New file `aggrequant/visualization/qc_plots.py`**
   - `plot_control_strip(csv_path, control_wells, output_path=None)` → Figure
   - Reads CSV → aggregates per well → maps to conditions → strip plot
   - Returns the matplotlib Figure (also saves if output_path given)
2. **Call from pipeline** `_generate_plots()` (alongside heatmaps)
   - Only if `control_wells` is non-empty
3. **Tests**: synthetic CSV + control_wells dict → verify figure axes, point counts

### 5. Fix deprecated `binary_dilation` warning in stardist.py

**Problem**: `skimage.morphology.binary_dilation` is deprecated since v0.26, removed in
v0.28. Line 164 of `stardist.py` triggers the warning every run.

**Fix**: Replace `binary_dilation(edge_mask)` with `dilation(edge_mask)` using
`skimage.morphology.dilation`. Since `edge_mask` is boolean, `dilation` on a boolean
array produces the same result (the default footprint is the same cross/diamond shape).
