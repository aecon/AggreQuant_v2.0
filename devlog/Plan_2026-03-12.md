## NN Module Redesign: Analysis and Architecture Selection

### What the backup code does

The `AggreQuant.bak/processing/aggregates.py` has two segmentation paths:

1. **`segment_aggregates_filters()`** — Already re-implemented cleanly as `FilterBasedSegmenter` in the current codebase. No action needed.

2. **`segment_aggregates_UNet()`** + `AggregateUnet()` — TensorFlow/Keras UNet inference:
   - Loads a `.keras` weights file
   - Pads image to next power of 2
   - Splits into 128x128 overlapping patches (stride=32)
   - Normalizes each patch to [0,1] by dividing by max
   - Runs batch prediction
   - Stitches patches back (averaging overlapping regions, discarding border strips)
   - Thresholds at p>0.7
   - Morphological cleanup (holes, small objects)

### Assessment of existing `nn/` module

The current `nn/` module is **comprehensive but untested** — well-structured scaffolding with every module one might want, but it has issues:

1. **Violates `__init__.py` convention** — `nn/__init__.py` does heavy re-exports with `__all__`, contradicting the "keep `__init__.py` as package markers only" rule
2. **Over-engineered for current needs** — `SEConvBlock`, `SEResidualBlock`, `CBAMConvBlock`, `CBAMResidualBlock` are never used; the `ModularUNet` composes `SEBlock`/`CBAM` via `EncoderBlock`/`DecoderBlock` instead
3. **Missing the inference pipeline** — No equivalent of the sliding-window patch/stitch logic from the backup
4. **18 variant combinatorial explosion** — The paper plan mentions "18 UNet variants" but testing 18 variants is impractical and scientifically weak (too many variables, no clear ablation story)

### Improvements over the backup code

1. **Normalization**: Backup uses `img / max(img)` — poor for microscopy. Current codebase uses percentile normalization — much better
2. **Stitching**: Backup discards border strips. Better approach: weighted blending in overlap regions
3. **Padding**: Backup pads to power-of-2 which wastes compute. Better: pad to nearest multiple of 16 (UNet constraint)

## Architecture Variant Selection

### Proposed variants 1-5: Classical UNet improvements (2015-2018)

For a methods paper, we want a **clean ablation story** — start simple, add one module at a time, show cumulative benefit:

| # | Name | Encoder | Decoder | Bridge | Skip | Attention | Rationale |
|---|------|---------|---------|--------|------|-----------|-----------|
| 1 | **Baseline UNet** | DoubleConv | DoubleConv | DoubleConv | Plain concat | None | Ronneberger 2015 reference |
| 2 | **ResUNet** | Residual | Residual | Residual | Plain concat | None | +residual connections (He 2016) |
| 3 | **Attention ResUNet** | Residual | Residual | Residual | Attention Gate | None | +attention gates on skips (Oktay 2018) |
| 4 | **CBAM Attention ResUNet** | Residual | Residual | Residual | Attention Gate | CBAM | +channel+spatial attention (Woo 2018) |
| 5 | **CBAM Attention ResUNet + ASPP** | Residual | Residual | ASPP | Attention Gate | CBAM | +multi-scale bridge (Chen 2017) |

**Why these 5:**
- Each adds exactly **one well-cited module** over the previous
- Creates a clean ablation table (Fig 2C in paper plan) where each row is one module addition
- All modules are proven for fine-structure segmentation (aggregates are small, boundary-sensitive)
- Deep supervision is orthogonal (training technique, not architecture) — test it as a training option on the best architecture, not as a variant

### Proposed variants 6-8: Post-2018 methods

All 5 variants above use modules from 2015-2018. Adding newer methods for a more complete story:

| # | Name | What's new vs previous | Year | Rationale |
|---|------|----------------------|------|-----------|
| 6 | **+ UNet++ skips** | Replace single skip connections with nested dense skips | Zhou et al. 2020 | Specifically designed for fine-grained objects; aggregates vary in size, dense skips capture multi-scale features better than single skips. ~800 citations/year. |
| 7 | **+ ConvNeXt encoder** | Replace ResidualBlock with modern ConvNeXt block (7x7 depthwise conv, LayerNorm, GELU, inverted bottleneck) | Liu et al. 2022 | Matches Vision Transformer performance with pure convolutions. Drop-in replacement for ResidualBlock — a modern encoder without transformer complexity. |
| 8 | **+ ECA (Efficient Channel Attention)** | Replace CBAM with ECA: lightweight 1D-conv channel attention, no reduction bottleneck | Wang et al. 2020 | Beats SE/CBAM with fewer parameters. Could replace CBAM as the attention module. Decision pending on whether to include. |

**Paper narrative:** *Classical UNet → residual → attention → modern convolutions → modern connectivity* — spanning 7 years of architectural progress in one clean ablation table.

UNet++ is particularly relevant because aggregates come in varying sizes (punctate vs diffuse), and dense skips are specifically designed for multi-scale object segmentation. ConvNeXt is the "what if we just modernize the convolution itself" question — a single-block swap, perfect for ablation.

### Transformer-based models (TransUNet, Swin-UNet): Feasibility assessment

**Question:** Can we use pretrained transformer weights for our grayscale microscopy images?

**Available pretrained weights:**

| Model | Backbone | Pretrained on | Download source |
|-------|----------|--------------|-----------------|
| **TransUNet** | ViT-B/16, ViT-L/16, R50-ViT-B/16 | ImageNet-21k (14M images) | [Beckschen/TransUNet](https://github.com/Beckschen/TransUNet) (Google Drive) |
| **Swin-UNet** | Swin-T | ImageNet-1k | [HuCaoFighting/Swin-Unet](https://github.com/HuCaoFighting/Swin-Unet) (Google Drive) |
| **CellViT** | ViT-B (custom), SAM-H | 104M histological patches (PanNuke) | [TIO-IKIM/CellViT](https://github.com/TIO-IKIM/CellViT) |
| **MedSAM** | ViT-B (SAM-based) | 1.57M medical images, 10 modalities | [bowang-lab/MedSAM](https://github.com/bowang-lab/MedSAM) |

**Grayscale handling:** All ViT-based models expect 3-channel RGB. Standard approach: duplicate grayscale channel 3× (G → RGB). Works in practice but wastes 2/3 of first-layer capacity.

**Small dataset concern (~19 annotated images):**
- ViTs lack CNN's convolutional inductive bias → more data-hungry → overfitting risk is significant
- Mitigation: Parameter-Efficient Fine-Tuning (PEFT/LoRA), freeze backbone and only fine-tune decoder, strong augmentation
- CellViT is most relevant (in-domain pretraining on histological patches), but it's designed for nuclei/cell segmentation, not aggregate segmentation
- Swin-UNet authors emphasize that pretraining is "very important" for pure transformer models

**Assessment:** Transformer models are **interesting but risky** for this paper:
- Adds significant implementation complexity
- 19 images is very small for transformer fine-tuning even with PEFT
- If transformer doesn't beat CNNs (likely with so little data), the result is still publishable ("transformers need more data")
- If it does beat CNNs, that's a strong finding
- **Recommendation:** Include as an optional variant if time permits. Not on the critical path.

## Inference Strategy: Sliding-Window vs Alternatives

### The question

Images are 2040×2040 pixels. The backup code uses sliding-window patch/stitch with 128×128 patches. Is this the best approach?

### Options evaluated

#### 1. Full-resolution inference (pad to 2048, single forward pass)

2040 is not divisible by 16 (2040/16 = 127.5), so pad to 2048 (next power of 2).

**Memory estimate for UNet [64, 128, 256, 512] at 2048×2048:**
- First encoder: 1 × 64 × 2048 × 2048 = 268M elements × 4 bytes ≈ 1 GB
- With all skip connections stored: ~4-8 GB for forward pass
- Feasible on 16-24 GB GPUs (RTX 3090/4090, A100)

| Pros | Cons |
|------|------|
| No stitching artifacts | High VRAM (~8-12 GB for forward pass) |
| Fastest (single pass) | Not feasible on small GPUs |
| Simplest code | Batch size limited to 1 |

**Verdict:** Best option if VRAM allows. Should be the **primary inference mode**.

#### 2. Tiled inference with overlap blending (improved sliding window)

Like the backup approach but with **Gaussian/cosine weighting** in overlap regions instead of hard border cropping. This is what **nnU-Net** uses.

| Pros | Cons |
|------|------|
| Works on any GPU | Slower (many forward passes) |
| No border artifacts (with proper blending) | More complex code |
| Memory-efficient | Normalization must be per-image, not per-patch |

**Verdict:** Best **fallback** for smaller GPUs. Use Gaussian weighting (not the backup's border-strip approach).

#### 3. Resize to fixed resolution (e.g., 2040→512 or 1024)

| Pros | Cons |
|------|------|
| Fast, simple | **Loses fine detail** — aggregates are small! |
| Low memory | Resizing back introduces interpolation artifacts |

**Verdict:** **Not suitable** for aggregate segmentation. Small aggregates (9+ pixels at native resolution) would become sub-pixel at 512×512.

#### 4. Overlapping tiles with center-crop stitching (backup approach)

The backup's approach: predict on overlapping patches, keep only the center region of each patch (discard borders where context is incomplete).

| Pros | Cons |
|------|------|
| Simple to implement | Wastes computation on discarded borders |
| No blending needed | Hard seams possible if stride is too large |

**Verdict:** Works but inferior to Gaussian blending. No reason to use over option 2.

### Recommendation

Implement **both** with automatic selection:
1. **Full-resolution** (default): Pad to multiple of 16, single forward pass. Use when VRAM is sufficient.
2. **Tiled with Gaussian blending** (fallback): For GPUs with <12 GB VRAM. Use 256×256 tiles (not 128—larger tiles give better context) with stride=128, Gaussian weighting in overlap regions.

Auto-detect: try full-resolution first, fall back to tiled if CUDA OOM.

1. Fix `__init__.py` files to be package markers only
